# 04 — UI

## Purpose

This document specifies the custom Next.js fit workspace that sits on top of the Hono REST surface in [`03-surfaces.md`](03-surfaces.md). The standalone MCP server and LangGraph Studio are the external-agent / visual-debug surfaces ([`03-surfaces.md`](03-surfaces.md)); the UI described here is the primary reviewer experience.

Upstream: [`01-entities.md`](01-entities.md), [`02-orchestration.md`](02-orchestration.md), [`03-surfaces.md`](03-surfaces.md). Downstream: [`05-evals.md`](05-evals.md) (the eval cockpit route renders the metrics defined there).

## Stack

- Next.js 15 App Router + React + TypeScript.
- Tailwind CSS + shadcn/ui for primitives (`Button`, `Dialog`, `Sheet`, `Tabs`, `Card`, `Toast`, `Badge`).
- TanStack Query v5 for client data (mutations, polling).
- Shared Zod schemas from `@rolegraph/shared` (package at `packages/shared/`) — the same schemas defined in [`01-entities.md`](01-entities.md). TypeScript types come from `z.infer`; no codegen step.
- Icon set: Lucide.

No global state library. TanStack Query is the cache; route params are the source of truth for the current selection.

## `/` — dashboard

- **Recent packets** table (columns: kind, title/filename, created_at, graph version).
- **Recent reports** table (columns: fit id, role title, candidate name placeholder, `coverageScore`, `confidence`, `createdAt`, status).
- **Actions:**
  - "Upload packet" button → `PacketUploadDialog` → `POST /packets`. After success, auto-navigates to `/roles/[id]` or `/candidates/[id]` depending on `kind`.
  - "New fit" button → `NewFitDialog` (pick a job packet + a candidate packet) → `POST /fit`. On `status === "running"` or `"paused"`, navigate to `/reviews` with the `runId` pinned; on `"completed"`, navigate to `/fit/[reportId]`.

## `/roles/[id]` — requirement graph viewer

Layout:
- **Header:** packet title, `version` badge, "Re-extract" button (`POST /roles/:id/extract?force=true`), "Use in fit" button.
- **List of requirements grouped by `kind`** (must / preferred / seniority / domain / location). Each uses `RequirementCard`.
- **Details drawer:** clicking a requirement opens `SpanHighlight` in a `DocumentViewer` scoped to the cited document(s).

## `/candidates/[id]` — evidence graph viewer

Same shape as `/roles/[id]`, with `EvidenceCard` and evidences grouped by `type` (role / project / technology / metric / scope / domain).

## `/fit/[reportId]` — fit workspace (centerpiece)

Three-pane layout with a drawer and a docked Q&A pane.

```
┌───────────────────────────────────────────────────────────────────────────────┐
│ Header: report id · coverageScore · confidence · mode toggle (Reviewer/Cand) │
├────────────────────┬──────────────────────────────┬───────────────────────────┤
│ Requirements       │ Coverage matrix              │ Evidence                  │
│ (RequirementCard)  │ (CoverageMatrix)             │ (EvidenceCard, filterable │
│ grouped by kind    │ cells: status color + label  │ by type)                  │
│                    │ click → lineage drawer       │                           │
├────────────────────┴──────────────────────────────┴───────────────────────────┤
│ Clarification questions (≤3) · Ambiguities                                    │
├───────────────────────────────────────────────────────────────────────────────┤
│ Q&A pane: input + cited answer / refusal (POST /query)                        │
└───────────────────────────────────────────────────────────────────────────────┘

Lineage drawer (opens on cell click):
┌─────────────────────────────────────────────────────────────────────────────┐
│ Requirement (text + kind + weight + citations rendered as SpanHighlight)    │
│ ──                                                                          │
│ Evidence(s) (for each linked ev_*: text + type + SpanHighlight in source)   │
│ ──                                                                          │
│ Rationale (LLM text, with evidence IDs linkified to the entries above)      │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Coverage matrix semantics

- One row per `Requirement`, ordered by `kind` then `weight` descending.
- The cell renders the match's `status`:
  - `strong` — green fill, label "Strong"
  - `weak` — amber fill, label "Weak"
  - `none` — red fill, label "Not evidenced"
  - `contradictory` — purple fill, label "Contradictory"
- Every color is paired with text; never color alone (see [Accessibility](#accessibility)).
- Cells show a confidence bar under the label.

### Mode toggle

- **Reviewer mode** (default): evidence map, ambiguity list, weak spots.
- **Candidate mode**: fit explanation, likely gaps, clarification questions.

Mode changes the right/bottom pane copy; it does **not** change the underlying data. No mode ever renders a hire/no-hire judgment; header shows a persistent `"Coverage, not candidate ranking"` helper chip.

### Q&A pane

- Input field + "Ask" button → `POST /query`.
- On success, renders the answer with each `TextSpan` citation rendered inline as a clickable `Citation` chip that scrolls the `DocumentViewer` to the span.
- On refusal (`refusalReason === "insufficient_grounding"`), shows an empty-state with the reason and suggested rephrasings.

## `/reviews` — HITL review queue

- **List** of paused runs (polls `GET /runs/:id` for each `runId` tracked by the client).
  - Row: `runId`, related `roleTitle`, `candidate` placeholder, `createdAt`, count of `riskFlags` grouped by `code`.
- **Selecting a row opens `ReviewDrawer`** (shadcn `Sheet`) with:
  - All `RiskFlag`s grouped by `code`, each with linked `refs` that open in a side preview.
  - A read-only preview of the draft `FitReport` (same components as `/fit/[reportId]`, with edit affordances on matches).
  - Action panel: `Approve`, `Edit`, `Reject` (radio) + `rationale` textarea (required for all actions) + patch builder when `Edit` is chosen.

### Risk flag → review queue behavior

Required by the verification checklist.

| `code` (from [`02-orchestration.md`](02-orchestration.md)) | Queue badge | Drawer call-out |
| --- | --- | --- |
| `NEGATIVE_CONCLUSION` | "Negative conclusion" (red) | Banner at top of drawer: "Aggregate coverage below 0.4. Review before finalizing a negative outcome." |
| `MISSING_MUST_HAVE` | "Missing must-have ×N" (red) | Each flag scrolls the matrix to the corresponding `req_*` row and highlights it. |
| `LOW_CONFIDENCE_MATCH` | "Low-confidence match ×N" (amber) | Each flag highlights the match's confidence bar in the matrix. |
| `PROTECTED_ATTRIBUTE_HIT` | "Protected attribute detected" (purple) | Banner: list of redaction categories hit, with a link to the `/traces/[runId]` redaction log. |
| `UNSUPPORTED_CLAIM` | "Unsupported claim ×N" (red) | Each flag links to the rationale text with the offending span underlined. |

### Patch builder

- Matches are the only editable entities in v1.
- For each match, the reviewer can change `status`, `confidence`, or add/remove `evidence_ids`.
- The patch builder emits an RFC 6902 `changes` array for the `POST /reviews` payload.

### Submission

- `POST /reviews` returns `resumedStatus`. If `"completed"`, toast + navigate to `/fit/[reportId]`. If `"rejected"`, toast and return to the queue. If `"running"` (after an edit that re-routes through `validateGrounding`), poll `GET /runs/:id` via the global `RunStatusToaster`.

## `/traces/[runId]` — trace inspector

- **TraceTimeline**: horizontal stages (`redactPii`, `extractRequirements`, ...) from `auditEvents`. Each stage shows `latencyMs`, `costUsd`, model id.
- Clicking a stage opens a side panel with:
  - `inputsHash` / `outputsHash` (raw payloads are not stored per [`01-entities.md`](01-entities.md); link to the persisted artifacts when applicable),
  - grounding report (if stage === `validateGrounding`),
  - redaction log (if stage === `redactPii`) — always rendered with categories, never raw redacted substrings.
- Deep link to `langsmith_run_url` at the top.

## Data fetching

### Reads

- Initial server-rendered reads use server actions (or fetch in a server component) and stream to the client.
- Client-side reads use TanStack Query with the server fetch's initial data as `initialData`.

### Mutations

All mutations go through TanStack `useMutation`. Query keys follow a stable shape:
- `["packet", packetId]`
- `["requirementGraph", packetId]` / `["evidenceGraph", packetId]`
- `["report", reportId]`
- `["run", runId]`
- `["trace", runId]`
- `["evalsSummary"]`

After `POST /reviews`, invalidate `["run", runId]` and `["report", reportId]`.
After `POST /fit`, invalidate `["run", runId]` and the dashboard query list.

### Polling paused runs

`RunStatusToaster` polls `GET /runs/:runId` with an adaptive interval:
- every 2s while `status == "running"`,
- every 5s while `status == "paused"`,
- stops on `completed` / `rejected` / `failed`.

On transition to `"paused"`, it surfaces a toast: *"Run {runId} is paused for review. Open queue →"* deep-linked to `/reviews?run={runId}`.

## Accessibility

- All status colors are paired with text labels (see coverage matrix legend and risk flag badge table).
- All interactive elements (match cells, requirement cards, evidence cards) are reachable by keyboard with visible focus rings.
- `CoverageMatrix` supports arrow-key navigation within the grid; `Enter` opens the lineage drawer; `Esc` closes it.
- Q&A pane: input has a visible label; citations are real anchors, not spans with click handlers.
- Document viewer: span highlights have accessible names (`aria-label`) that include category (`requirement`, `evidence`, `redaction`).
- Color contrast ≥ WCAG AA for all status colors against their backgrounds.

## Error handling

- All mutations show a shadcn `Toast` on failure, with the `code` and `detail` from the RFC 7807 problem+json.
- Validation errors on `POST /reviews` highlight the offending match in the patch builder instead of only showing a toast.
- If `GET /runs/:id` returns `status === "failed"`, the `RunStatusToaster` renders a persistent error toast with a "View trace" link to `/traces/[runId]`.
